Designing in the void

May 23, 2024
3 min read

The system consists of interdependent software elements.The organizational structure of these program elements together with their unifying relationships is called architecture.On diagrams depicting such systems, individual program elements and servers are often simplified in the form of rectangles connected by arrow keys.

One small arrow may mean: “Synchronous SOAP-XML Request/Respond via HTTP”.There is too much information for one element of the chart.There is usually not enough space for complete writing, so we mark the arrow with “XML through HTTP” (from the point of view of internal implementation) or “Search by product code” (for external users).

The arrow that connects programs looks like a direct contact, but it is not.The vacuum between rectangles is filled with hardware and software components.In particular, the following can be located:

  • Network maps
  • Network Switches
  • Firewalls
  • Intrusion Detection and Prevention Systems (IDS/IPS)
  • Brokers or message queues
  • XML Conversion Mechanisms
  • FTP Servers
  • Zone tables
  • SoNET Rings
  • MPLS Gates
  • Highway lines
  • Oceans
  • Fishing trawlers damaging bottom cables

Between program elements A and B there are always four to five computers running package switching, traffic analysis, routing, threat analysis, etc. And you, as an architect connecting these program modules together, must take into account the existence of this intermediate environment.

One day I saw a arrow with the inscription “Implementation.“One server was installed in my client’s company, the other was located in a completely different location.This crucial client arrow triggered a chain of events that resembled the Mouse Run game more than a single interface. Messages were transmitted to brokers, storing them in files that were loaded on FTP… etc.This “interface” included more than 20 stages.

It is necessary to understand well what static and dynamic load falls on each arrow.Next to “SOAP-XML over HTTP” by the arrow, it would also be worth writing: “For each Http request one call is accepted, for each HTTp answer one response is returned.Up to 100 requests per second are expected, and 99.999% of the requests should be delivered in less than 250 ms.”

But that’s not all you need to know about this arrow:

  • What if the calling party calls too often?How should the recipient act – ignore requests, polite refusal or, if possible, try to process them?
  • What should the calling party do if the processing of the request has taken more than 250 ms?Do you want to try again?Wait a minute?Decide that there was a failure on the recipient side and continue without this function?
  • What happens if the calling party has sent a request for version 1.0 and received a response in version 1.1?What if HTML was obtained instead of XML?Or an MP3 file instead of an XML file?
  • What happens if one side of the interface becomes temporarily inaccessible?

The answers to these questions are the essence of designing “in empty space” diagrams.

There are no solutions for all cases of life Creating architecture as the art of balance
